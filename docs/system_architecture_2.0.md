# 系统架构 2.0: 量化交易平台与策略技能化 (Runner 2.0)

## 1. 设计核心理念

从架构 2.0 开始，系统正式从“单策略适配”进化为“平台-插件（Skill）”架构。

- **平台 (Runner/Engine)**：作为操作系统，提供数据总线（BUS）、订单执行（Executor）、Dashboard 渲染等公共基础设施。
- **策略 (Strategy/Skill)**：作为标准化插件，只运行核心业务逻辑。策略不再需要了解底层数据是如何缓存或预热的。

## 2. 策略标准合约 (Standard API)

所有策略必须继承 `BaseStrategy` 并遵守以下标准接口契约。

### 2.1 数据摄入与预热
- `warmup(self, data_list: List[MarketData])`: 
  - **职责**：策略在此处自主处理历史数据，初始化指标。
  - **解耦点**：引擎不再强行推入 `_update_buffer`，策略内部是否维护缓存由策略自己决定。

### 2.2 决策执行
- `on_data(self, data: MarketData, context: StrategyContext) -> List[Signal]`: 
  - **职责**：接收来自总线的 Tick 或 Bar 数据，输出标准交易信号。
  - **高频支持**：支持 2 秒频率的 Tick 预览计算。

### 2.3 UI 协议规范 (V6.0 预研)
- `get_ui_manifest(self) -> Dict[str, Any]`: 
  - **职责**：策略自主定义 Dashboard 所需组件。
  - **动态性**：支持 6.0 神经网络策略所需的“热力图”、“权重分布”等非标准化视图。

## 4. 核心组件说明

### 4.1 引擎层 (Engines)
这一层决定了策略以何种方式运行（模拟时间 vs 现实时间）。

*   **BacktestEngine (`engines/backtest.py`)**：
    *   **职责**：在历史数据上回溯，验证策略逻辑。
    *   **特点**：高效率事件驱动，计算回撤、胜率等收益指标。
*   **LiveEngine (`engines/live.py`)**：
    *   **职责**：在真实市场中驱动策略运行。
    *   **特点**：处理实时数据流、预热历史 K 线、维护 WebSocket 连接并同步 Dashboard 状态。

### 4.2 执行层 (Executors)
这一层决定了信号最终的去向。

*   **PaperExecutor (`executors/paper.py`)**：
    *   **职责**：本地内存模拟撮合。
    *   **特点**：无需联网，支持滑点和延迟模拟，常用于回测或无风险模拟测试。
*   **OKXExecutor (`executors/okx.py`)**：
    *   **职责**：通过 API 连接 OKX 交易所。
    *   **特点**：支持模拟盘 (Demo) 和实盘 (Live) 切换，负责真实资产的下单与对账。
